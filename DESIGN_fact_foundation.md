## 0. 现状调研（已确认，两套机制已存在但有断裂）

- ✅ **增量更新**（knowledge/updater.py）：git push webhook → 变更文件事件 → 只重索引变更文件（Layer A 结构 / B 语义）+ 去重。机制符合"项目变才更新、不频繁"。
- ✅ **模板复用**（image_builder.py）：`fingerprint = v{BUILDER_VERSION}-{deps_hash}-{src_hash}`，指纹未变复用旧模板，依赖/源码/构建逻辑变了才重建。符合"原项目没改复用、加新东西才重打"。
- ❌ **断裂1（核心）**：worker 在沙箱建新文件 → pull-back 到本地，但【不 git commit/push】→ updater 收不到事件 → 知识库不知新文件 → 下个任务核验"BizDevice.java 在不在"误判"不在"（其实上轮建了）。**事实库滞后于实际产出**。
- ❌ **断裂2**：核验目前是静态单次（查一次知识库），无可信度概念，索引漏一个文件就误杀。
- ✅ **可复用**：L6 衰减（0.95）、norms_store/behavior_store 已有"分数随时间/反馈调整"的底子，可扩展成事实可信度。

## 1. 三个根本问题与方案

### 问题A：事实库滞后（断裂1）——worker 产出不回灌知识库
**根因**：产出 pull-back 后知识库不更新。
**方案**：worker 子任务 DONE 后（产出已落本地 + L1 通过），把【本子任务新建/修改的文件】直接喂给 knowledge updater 做增量索引（不依赖 git push webhook，本地产出直接触发）。这样下个任务/下个子任务的核验能看到最新事实。
- 轻量：只索引变更的几个文件（updater 本就支持文件级增量），非全量。
- 时机：子任务 DONE 后异步触发，不阻塞主流程。

### 问题B：静态核验易误杀——引入动态事实可信度
**方案**：事实（如"文件X存在/不存在"、"表Y有/无"）带可信度分数，证实加分、证伪减分（复用 L6 衰减式机制）：
- 知识库索引到 → +；worker 沙箱实际确认 → ++（沙箱是 ground truth，权重高）；worker 发现与知识库相反 → 给知识库记录减分 + 回灌修正。
- tech_design 核验时：高可信度的"不存在"才触发澄清；低可信度（可能索引漏）→ 降级为"方案里加警告，让 worker 沙箱实地确认"，不硬打断。
- **避免误杀的关键**：沙箱实地结果 > 知识库索引（索引可能滞后，沙箱是真相）。

### 问题C：预处理/模板重建时机——已有机制，补"产出后增量"和"防频繁"
- 复用现有 fingerprint 模板复用 + git 增量。
- 补：worker 产出回灌（问题A）顺带让 knowledge 增量更新，**但模板不必每次重打**——模板只在 deps/工具链变（pom.xml 改）时重建（fingerprint 的 deps_hash 变才变），新增业务文件不改依赖 → src_hash 变但可配置"业务文件变不强制重打模板"（模板装的是工具链/依赖，不是业务源码）。
- 防频繁：增量索引按"子任务 DONE"批量触发（非每文件），模板重建只认 deps_hash。
