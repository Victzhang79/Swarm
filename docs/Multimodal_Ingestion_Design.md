## 0. 需求来源（用户原话要点）

1. 需求不一定是文字——支持多图/PDF/DOCX/MD：文本类确定性解析优先，仅图片/扫描件走多模态。
2. 图片由 AI 理解 → **人工确认**（除非用户勾选"模型自行确认"选项）。

   因为用户会改模型，不能在代码里凭记忆/写死（如规划预算写死 150k）。
6. 文档解析用轻快依赖，先不上复杂的（marker）。

8. **任务列表 = 需求池**：任务建立后不一定立即执行，可"入池"待执行；摄取产出的需求草稿持久化可回看。

# A. 模型能力探测与注册（基础设施）

## A.1 目标
配置一个接入点(provider)时，对其下模型探测真实能力，存库；用户改模型可重新探测。
路由、上下文预算、多模态选型全部读库，消除写死常量。

## A.2 探测内容（每模型一条 ModelCapability 记录）
| 字段 | 怎么探测 | 兜底 |
|---|---|---|
| `model_id` | `GET {base_url}/v1/models` 列出 | 用户手填 |
| `context_window` | **分层探测**(见 A.2.1)：models 字段 → vLLM max_model_len → 错误消息解析 → 启发式默认 | 启发式默认 + 标注"未探明" |
| `supports_multimodal` | 发一条含 `image_url` 的最小消息，正常响应=True，报"不支持图像/invalid"=False | 默认 False |
| `gen_speed_tps` | 发固定小 prompt，测 tokens/秒 | 0=未测 |
| `kind` | 来自 provider(local/cloud) | — |
| `probed_at` / `source` | 探测时间戳 + 来源(probed/parsed/manual/default) | — |

### A.2.1 context_window 分层探测（核心，全走探测但不伤服务端）
按成本从低到高，命中即停：
1. **`GET /v1/models`** 读 `context_window` / `max_model_len` / `max_context_length` 等字段
   （OpenAI/部分 vLLM 返回；很多端点没有 → 进下一步）。
2. **vLLM `/v1/models` 详情或专有字段**（部分 vLLM 在 model 对象里暴露 `max_model_len`）。
3. **错误消息解析（低成本拿真值的巧法）**：发一个**故意超长**的最小请求，从 400 报错里 parse
   `maximum context length is (\d+)` / `max_model_len.*?(\d+)` 等模式——一次请求即拿真值，
   比二分试探便宜得多，且服务端只是拒绝(不真正分配超大上下文，不 OOM)。
4. **启发式默认**：按模型名/规模给保守值(本地小模型 32k，大模型 128k)，标 `source=default` +
   UI 提示"未探明，建议人工确认"。
- **明确不做**：二分试探上限(发递增真实长 prompt)——伤服务端、占显存、拖慢在跑的模型。

> 多模态/速度探测低成本可靠；context_window 靠上述 4 层，绝大多数端点能在 1-3 步拿到真值或近似。

## A.3 存储
- 新表 `model_capabilities(provider_id, model_id, context_window, supports_multimodal, gen_speed_tps, kind, probed_at, source)`
  source ∈ {probed, manual, default}，区分数据来源（探测/人工/默认兜底）。
- 或复用 config 的 JSON 字段（轻量）。倾向独立表（可重探、有时间戳、可审计）。

## A.4 探测触发（A3 用户确认）
- **配置/变更接入点后自动探测一次**——但**绝不阻塞保存**：
  - 保存接入点 → 立即返回成功（用户无感、不卡顿、不会"保存不上"）。
  - 后台**异步**探测该 provider 全模型 → 写库 → 前端轮询/SSE 显示「探测中…」→ 完成刷新能力展示。
- 也提供手动「重新探测」按钮（换模型/重启推理服务后用）。
- API：`POST /api/models/probe?provider_id=X`（触发，立即返回 job）+ `GET /api/models/capabilities`（读）+ `PUT`（人工修正）。
- 失败/未探测 → 用启发式默认，UI 标注"未探测/默认值"，提示重探或手填。

## A.5 消费方（消除写死）
- **上下文预算（A4 用户确认，诚实分步）**：
  - **现状诚实评估**：规划的 `est_context_tokens` 目前是启发式(难度基线+文件数×6k)，是**粗估非实测**。
    拿粗估 × 0.75 的"安全感"是假的——0.75 系数的前提("上下文控制较准")当前**不成立**。
  - **第一步(本轮)**：预算 = `min(实际模型 context_window × 0.75, 启发式兜底)`。用探测到的真实窗口设上限，
    但不假装预估精确，保留保守兜底。写死常量降级为"无能力数据时兜底"。
  - **第二步(后续债，记入 TECH_DEBT)**：`est_context_tokens` 升级为 **tokenizer 实算**(系统prompt+知识库+
    代码+历史的真实 token) + **执行后回写校准**(记录子任务实际用量，实测>预估则调高估算)，形成反馈闭环。
    那时 0.75 才名副其实。**不在本轮假装做到"准确上下文控制"**。
- **多模态选型**：摄取层从能力库筛 `supports_multimodal=True` 的模型，而非读写死的 routing_multimodal。
- **路由**：tier 路由可参考速度/上下文做更优选择（后续）。

## A.6 ⚠️ 工程注意
- 探测有副作用（真实 API 调用、花 token/算力）——必须用户显式触发，不在启动时自动全探。
- 本地推理服务(vLLM/Ollama)的 /models 格式各异 → 解析要容错。
- context_window 试探上限会发超长请求，可能 OOM 服务端 → **默认不做主动试探**，只人工填+默认。
- 探测结果有时效（换模型/重启服务会变）→ 存 probed_at，UI 显示"X 天前探测"，支持重探。

# B. 多模态需求摄取层（消费 A）

## B.1 架构
```
任务创建(文字 + 上传文件[图片/pdf/docx/md])  ← 入口：大小+格式+安全限制
  ▼
INGEST 摄取层(brain/ingest.py, analyze 前置)
  ├─ 按类型分流：
  │   ├─ md/txt → 直读
  │   ├─ pdf(文本层) → PyMuPDF 抽文字 → 摘要
  │   ├─ docx → python-docx 抽文字 → 摘要
  │   ├─ pdf(扫描/无文本层) → 渲染图 → 多模态模型(从能力库选)
  │   └─ 图片 → 多模态模型理解(标注"AI 理解，待确认")
  ├─ 多文件合并成「需求草稿」
  ▼
需求草稿 → ANALYZE → CLARIFY(确认图片理解) → ASSESS → ...(现有链路不变)
```

## B.2 防幻觉（Q2）
- 图片理解结果标注 `source=ai_vision, confirmed=False`。
- 默认**强制进 clarify 确认**："我从图理解到 X，对吗？"
- 用户可勾选「模型自行确认」选项 → 跳过人工确认（信任模型）。该选项存任务/全局配置。

## B.3 安全限制（Q4，入口重点）
- 格式白名单：仅 png/jpg/webp/pdf/docx/md/txt。
- 单文件大小上限（默认 20MB，可配）+ 总大小上限 + 文件数上限。
- 上传路径隔离（任务专属目录，不可路径穿越 `../`）。
- PDF 解析在受限环境，禁用 JS/外部资源加载（PyMuPDF 默认不执行 JS，确认配置）。
- 解析超时 + 临时文件用后清理。
- MIME 校验（不只信扩展名）。

## B.4 上下文预算（消费 A.5）
PDF 几万 token → 摘要/分块，按摄取目标模型(从能力库取 context_window)控制注入量，不塞爆 analyze。

## B.5 需求池模式（Q8）
- 任务创建时可选「立即执行」或「仅入池」(draft/pending)。
- 入池任务停在"需求成型"(经摄取+可选澄清)，不进 dispatch；用户后续手动「执行」触发。
- 摄取草稿 + 澄清结果持久化(复用 planning_artifacts 思路)，详情页回看原始文件+理解结果。
- 任务列表区分"待执行(池)"vs"执行中/已完成"。
