1. 开发者通过CLI/WebUI提交任务 ↓ 2. API层校验 → 入队（Redis Queue）→ 检查互斥锁 ↓ 3. Brain拿到任务 → 读取L1/L2/L4/L5/L6 → 判断复杂度 → 拆解DAG ↓ 4. Brain按DAG调度Worker → 并行/串行执行 ↓ 5. Worker在Docker容器中：拉代码→编码(内循环)→验证→产出diff ↓ 6. Brain 编排层

LangGraph状态机，系统的大脑，不写代码只做决策

Brain 核心流程

1️⃣ 需求理解

• 解析开发者意图

• 补充项目上下文（从L4知识库检索）

• 判断任务复杂度 → 路由策略：

简单（改配置/加字段）→ 单Worker直接执行
中等（单模块功能）→ 2-3 Worker串行
复杂（跨模块Feature）→ 多Worker并行+合并
超复杂（架构变更）→ 先出方案让人确认
2️⃣ 任务拆解

• 生成子任务DAG（带依赖关系）

• 为每个子任务指定：范围、约束、验收标准

• 检索L5错题集 → 注入"不要犯的错"

• 检索L6成功集 → 注入"可参考的范例"

3️⃣ 执行调度

• 按DAG拓扑序调度Worker

• 无依赖的子任务并行执行

• 有依赖的等上游完成后启动

• Worker失败 → 重试(最多2次) → 换模型重试 → 上报人工

4️⃣ 产出审查

• 合并所有Worker产出

• 冲突检测与解决

• 整体一致性检查（接口对齐、import完整等）

• 自动跑Lint + 编译验证

• 生成变更摘要给开发者

Brain 记忆系统 L0-L6

L0 会话元数据 内存，用完即弃 内存
L1 用户档案卡 PostgreSQL，精确KV，覆盖更新。每个开发者的偏好、风格、常用模块 PostgreSQL JSON
L2 近期任务摘要 PostgreSQL，滚动50条，只存摘要。避免重复工作、发现关联 PostgreSQL
L3 滑动窗口 内存，当前任务上下文，超限自动压缩旧消息 LangGraph State
L4 长期知识库 Qdrant + 原文，代码结构/文档/接口定义。详见知识库4层设计 Qdrant + PG
L5 错题集 PG + 向量索引，错误模式+教训，带衰减。被拒绝时提取 PG + 向量
L6 成功模式集 PG + 向量索引，最佳实践+范例代码。被接受时提取 PG + 向量

任务拆解策略

4大拆解原则 + 共享契约机制

原则1：每个子任务的文件范围必须明确且最小化
❌ 错误拆法："实现用户列表排序功能"（太模糊，Worker不知道动哪个文件） ✅ 正确拆法： 子任务1："在UserQueryDTO.java中新增sortField和sortOrder字段" 子任务2："在UserMapper.xml的selectPage中加动态排序SQL" 子任务3："在UserController.getUserList中接收并传递排序参数"
原则2：子任务之间的边界不能有歧义

每个子任务产出的"接口契约"要在拆解时就定好。Brain承担"架构师"角色，所有共享接口由Brain统一定义，Worker不做设计决策只做实现。

原则3：独立可验证

每个子任务完成后必须能独立验证对错。例如：子任务1验证DTO编译通过、子任务2验证XML语法正确、子任务3验证Controller编译通过。

原则4：宁可多拆不要少拆

一个子任务只改1-3个文件，超过3个文件的子任务大概率要继续拆。原因：Worker上下文有限，塞太多文件容易顾此失彼。

拆解示例 + 共享契约

需求："给用户列表加排序功能"

Brain拆解并定义契约：

// 共享契约（由Brain统一定义，写入所有相关子任务的context） { "new_dto_fields": [ {"name": "sortField", "type": "String", "default": "createTime"}, {"name": "sortOrder", "type": "String", "default": "DESC"} ], "validation_rule": "sortField只允许: createTime, username, email", "mapper_param_style": "#{sortField} #{sortOrder}", "controller_param_style": "@RequestParam(required=false)" } // 子任务A（可并行）：修改 UserQueryDTO.java scope.writable: [UserQueryDTO.java] context: 按契约添加两个字段，加@Schema注解 验证：编译通过 // 子任务B（可并行）：修改 UserMapper.xml scope.writable: [UserMapper.xml] scope.readable: [UserQueryDTO.java] ← 知道字段名即可 context: 按契约在selectPage的SQL末尾加动态ORDER BY 验证：XML语法校验通过 // 子任务C（依赖A）：修改 UserController.java + UserService.java scope.writable: [UserController.java, UserService.java] scope.readable: [UserQueryDTO.java] context: 按契约在Controller接收参数，set到DTO中透传 验证：编译通过 // 执行顺序： A ─────┐ ├──→ C（C依赖A的DTO定义完成） B ─────┘ （B和A/C并行，因为B只需要知道字段名，不需要真实文件）

跨文件协调三层保障

第一层：拆解时预防（Brain负责）

Brain在拆解时就把"接口契约"定死。所有共享接口由Brain统一定义，Worker只管按规定写。

关键：Brain承担"架构师"角色，所有共享接口由Brain统一定义，Worker不做设计决策，只做实现。

第二层：执行时约束（Worker隔离）
文件锁定规则： scope.writable: ["UserMapper.xml"] → 可以修改 scope.readable: ["UserQueryDTO.java"] → 只能读，参考结构 其他文件：不可见 如果Worker发现需要修改scope外的文件 → 不能擅自改 → 上报Brain → Brain决定是否扩展scope或新开子任务
第三层：合并时验证（Brain审查）

所有Worker完成后，Brain做"集成审查"：

import一致性Worker-A新增的类，Worker-B是否正确import了
类型一致性字段类型在所有文件中是否一致
命名一致性同一概念在不同文件中是否用了同一个名字
调用匹配方法签名和调用处参数是否匹配
编译验证合并后整体编译是否通过

如果发现不一致 → Brain自动生成修复子任务 → 单独Worker执行修复

合并冲突处理

三种冲突场景
场景1：Worker之间的冲突（内部冲突）Brain拆解有问题，应在拆解阶段就避免（PlanValidator检查）
场景2：蜂群产出和主分支的冲突（外部冲突）Worker在t0拉了代码，t1完成时主分支已有新提交 — 最常见场景
场景3：多个蜂群任务之间的冲突两个任务同时修改同一模块 — 通过任务队列的互斥锁避免
冲突处理流程
Worker产出diff │ ▼ Step 1: 尝试合并 (git apply --check) │ ┌──┴──┐ 成功✅ 失败❌ │ │ ▼ ▼ 直接 Step 2: 分析冲突类型 应用 │ ┌──┼──┐ ▼ ▼ ▼ 无关变更 相邻变更 同区域变更 (不同区域) (上下文偏移) (真冲突) │ │ │ ▼ ▼ ▼ 3-way 重新基于 Brain分析 → merge 最新代码 选择策略： 自动解决 让Worker a. 自动合并（语义不冲突时） 重新生成 b. 重新执行子任务（基于最新代码） c. 上报开发者决定
冲突预防措施
缩短执行时间Worker并行执行，任务越快完成diverge越少
执行前rebaseWorker开始前先拉最新代码
最小化修改范围Worker只修改必要的行，不做格式化、不动无关代码
同模块互斥同一模块同一时间只有一个蜂群任务在执行（Redis分布式锁）
快速失败合并前检查主分支是否有新提交，变更文件被改了就直接重跑

Base Layer（月更） OS + JDK/Node/Python + 基础工具 (git, tree-sitter, linter) │ ▼ Project Layer（周更/代码变更时） 项目依赖已下载 (mvn dependency已缓存 / node_modules) 项目最新代码已clone │ ▼ Task Layer（每次任务时） checkout到最新commit 创建工作分支 注入任务上下文文件

• Worker启动不需要从头clone + 下载依赖

• 从拿到子任务到开始编码：<10秒（只需git pull + checkout）

• 全量编译耗时大幅减少（增量编译）

• 10个项目各保持1个预热容器待命

• 任务来了直接fork一份使用

Worker 执行引擎详细

输入

• 子任务定义（目标、约束、验收标准）

• 文件scope（可读/可写）

• 契约信息（共享接口定义）

• 上下文代码片段（相关文件内容）

• 错题警告 + 成功范例

执行阶段
Phase 1: 定位（<5秒）读取scope内文件 → 理解当前代码结构 → 确定修改点
Phase 2: 编码（10-60秒）生成代码修改 → 写入文件
Phase 3: L1验证（10-120秒）编译 → lint → 相关单测 → 自检。失败回到Phase 2（最多3轮）
Phase 4: 产出生成diff → 附带变更说明 → 置信度评分 → 上报Brain
输出

• patch文件（git diff格式）

• 变更说明（改了什么、为什么这么改）

• 置信度（high/medium/low）

• 验证结果（L1各项是否通过）

• 执行日志（用于debug）

更新：Git push webhook → 只重新解析变更的文件（增量）

Layer B：语义检索层（模糊匹配，用向量）

数据来源：代码文件 + 注释 + 文档 | 存储：Qdrant | Embedding：bge-m3

索引粒度（关键决策）： ❌ 不按文件整体索引（太粗，噪音大） ❌ 不按单行索引（太细，丢失上下文） ✅ 按"语义单元"索引： - 一个完整方法/函数 - 一个类的签名 + 所有方法签名（不含方法体） - 一段连续注释/文档块 - 一个配置文件段落 每个chunk附带元数据： file_path, module, layer, class_name, method_name

用途：开发者说"类似订单导出的功能"，能找到相关代码

Layer C：项目规范层（人工维护 + 自动提取）

存储：PostgreSQL + Markdown原文

• 架构分层规范（Controller/Service/Mapper各层职责）

• 命名约定（URL风格、类名后缀、变量命名）

• 异常处理规范（全局异常 vs 业务异常）

• 数据库规范（字段命名、索引策略、分页方式）

• Git规范（分支命名、commit message格式）

• 从.editorconfig/.eslintrc/pom.xml自动提取的约束

注入方式：每次Brain拆任务时，相关规范作为system prompt的一部分。不需要检索，直接按项目ID全量注入（规范文档通常不大）

Layer D：历史行为层（自动积累）

存储：PostgreSQL

• 最近100条MR的摘要（谁改了什么，为什么）

• 高频修改文件排名（哪些文件经常一起被改）

• 模块间调用热度（哪些模块经常联动）

用途：Brain判断"改这个接口，是不是下游也得跟着改"，预测潜在影响范围

知识检索流程示例

以"给用户列表接口加按注册时间排序"为例，展示4层知识如何配合

第一步：精确定位（Layer A 结构索引）
关键词提取："用户列表接口" → 搜索symbol_index 命中：UserController.getUserList() UserService.queryPage() UserMapper.selectPage() 沿依赖图扩展： getUserList → 调用了 userService.queryPage queryPage → 调用了 userMapper.selectPage selectPage → 关联 UserMapper.xml 中的SQL 产出：确定需要查看/修改的文件列表（精确）
第二步：语义扩展（Layer B 向量检索）
Query: "列表接口排序功能实现" 检索结果： - OrderController.getOrderList() → 里面有排序实现（参考范例） - ProductService.queryWithSort() → 另一个排序实现 用途：给Worker提供"项目里类似功能是怎么做的"参考
第三步：规范注入（Layer C 直接读取）
项目: user-service 注入规范： - "排序字段必须有数据库索引" - "分页接口统一使用PageResult包装" - "新增查询参数需要在DTO中加字段，不允许用Map接收"
第四步：历史参考（Layer D + L5/L6）
错题集命中： "上次给订单列表加排序时，忘了在Mapper.xml中加动态SQL判断， 导致sortField为空时SQL报错" 成功模式命中： "商品列表排序的标准写法：Controller接收排序参数 → Service透传 → Mapper用<if>动态拼接ORDER BY" 历史行为： "UserMapper.xml 和 UserMapper.java 最近3次MR都一起改的" → 确认这两个文件需要同时修改

向量检索 Rerank 优化

问题：向量检索经常召回不相关的内容怎么办？

解决方案：检索后rerank

原始检索 top_20 → Reranker模型精排 → 取top_5 Reranker选择： - bge-reranker-v2-m3（本地部署，延迟低） - 或用LLM做：把20条结果让Brain评估哪些真正相关 额外优化： - 元数据过滤：只搜同项目、同模块（如果能确定模块的话） - 时间权重：最近修改的文件权重更高 - 共现过滤：参考Layer D的"经常一起修改的文件"

知识库与Brain状态机的交互

Brain执行任务时
ANALYZE阶段 → 读取知识库（Layer A/B/C/D + L5/L6） 此时知识库必须是最新的 保障措施：ANALYZE开始前先检查： last_push_time = await git_service.get_last_push_time(project) last_index_time = await kb_service.get_last_update_time(project) if last_push_time > last_index_time: await kb_service.wait_for_update(project, timeout=60)
任务完成时 — 闭环

LEARN_SUCCESS → 写入L6 → 产出被merge到主分支 → 触发Git push webhook → 知识库自动增量更新 → 下次任务时能看到最新代码

闭环：做任务 → 产出代码 → merge → 知识库更新 → 下次任务更准

Worker执行时也可能需要检索知识库

Worker在容器内执行时，如果需要查"某个接口的定义在哪" → 通过工具调用Brain提供的检索API → 不是Worker直接访问知识库，而是通过统一的检索服务

原因：Worker容器是隔离的，不应直接访问数据库。统一检索服务可以做缓存、权限控制、日志。Brain可以在检索结果上追加上下文。

L1 验证 — Worker内验证

单文件/单子任务级别 | 执行环境：Worker的Docker容器内 | 耗时：几秒到几十秒

L1.1 语法验证
Javajavac 编译单文件（或 mvn compile -pl 单模块）
JS/TStsc --noEmit
Pythonpython -m py_compile
XMLxmllint --schema

通过条件：零编译错误 | 失败处理：读取错误信息 → 让Worker修复 → 重新验证（最多3轮）

L1.2 静态检查

Lint: checkstyle / eslint / pylint（只检查修改的文件）
规范：import是否多余、命名是否符合规范

通过条件：无error级别问题（warning可忽略）

L1.3 单元测试（如果有相关测试）
策略：只跑和修改文件相关的测试用例，不跑全量 定位方法： 修改了 UserService.java → 找到 UserServiceTest.java → 只跑 UserServiceTest 通过条件：相关测试全部绿色 失败处理： - 新增功能导致旧测试断言失败 → Worker分析是否需要更新测试 - 逻辑错误 → Worker修复代码 - 测试本身不合理 → 标记为需要人工确认
L1.4 自我审查（LLM自检）
Worker完成编码后，用一个独立prompt自检： "检查你的修改是否满足以下条件： 1. 是否完整实现了子任务目标？ 2. 是否有遗漏的边界情况？ 3. 是否违反了约束条件？ 4. 修改的代码风格是否和周围代码一致？" 如果自检发现问题 → 自行修复后再次自检

L2 验证 — 集成验证

合并后的完整项目级别 | 独立Docker容器 | 耗时：1-5分钟 | 触发：所有Worker产出合并后

L2.1 全量编译

Java: mvn compile -DskipTests（整个项目编译）| 前端: npm run build
目的：验证跨文件的import、类型引用、接口调用是否一致。这一步能发现Worker之间"对不上"的问题。

L2.2 相关模块测试

跑涉及修改的模块的全部测试（不是全项目，而是受影响模块）
mvn test -pl user-service
如果项目有测试速度问题：只跑被修改类的直接测试+一跳关联测试

L2.3 接口一致性校验

如果修改了API接口：检查Swagger/OpenAPI文档是否同步更新、检查上下游调用方的参数是否匹配、如果是微服务间接口 → 检查Feign Client定义是否一致

失败处理

• 编译失败 → Brain分析错误 → 指派修复子任务给Worker
• 测试失败 → Brain判断是代码bug还是测试需要更新
• 2轮修复后仍然失败 → 标记为需要人工介入

L2 集成验证容器流程
1. 基于最新主分支创建验证分支 2. 依次apply所有Worker的patch 3. 全量编译 4. 运行受影响模块的测试 5. 通过 → 产出最终合并diff 失败 → 回到Brain分析

L3 验证 — 测试环境部署验证

利用团队已有的CI/CD | 耗时：5-15分钟 | 可选执行

✅ 需要L3

• 修改了数据库Schema（要验证migration能跑通）

• 修改了外部集成（支付、短信等）

• 修改了中间件配置（Redis、MQ的连接方式）

• 涉及多服务联调的功能

❌ 跳过L3

• 纯逻辑变更（L2测试已覆盖）

• 前端UI修改（不涉及后端联调）

• 配置项修改（改个超时时间）

L3验证方式

不在蜂群自己的容器里做，而是利用团队已有的CI/CD

蜂群要做的： - push代码到临时分支 - 通过GitLab API监听pipeline状态 - pipeline结果回流到任务状态 - 失败日志提取 → Brain分析 L3验证流程： 1. 将蜂群产出的分支部署到测试环境（通过CI/CD pipeline触发） 2. 跑集成测试/E2E测试（项目已有的 or 蜂群生成的smoke test） 3. 健康检查（服务启动、数据库连接、依赖服务调用） 4. 回滚（验证完毕后自动回滚测试环境到之前的版本）

⚠️ 重要约束

• 测试环境验证是非阻塞的：开发者可以不等L3就先看diff
• L3结果作为MR的附加信息展示："测试环境验证通过/失败"
• 不自动merge，最终决定权仍在开发者

风险评估

LLM幻觉导致错误代码

L1/L2验证无法覆盖所有逻辑缺陷。
缓解：L5错题集持续积累、L2验证覆盖核心路径、最终必须开发者审查、高风险任务强制confirm。

任务拆解质量不稳定

LLM对复杂需求拆解可能不合理。
缓解：VALIDATE_PLAN自动校验 + 最多2次重试 + 超复杂人工确认 + L6成功模式参考。

开发复杂度高，MVP周期长

涉及LangGraph、Docker管理、多数据库、实时通信等众多技术栈。
缓解：分阶段交付，Phase0先验证单Worker核心链路。

LLM API成本可能超预期

重试、多轮修复可能导致单任务token消耗远超预估。
缓解：设置单任务token上限、Worker最大轮次限制、监控预警。

知识库增量更新不一致

Webhook丢失、解析异常等导致索引和实际代码不一致。
缓解：每日ConsistencyChecker + 每周全量重建 + ANALYZE前检查时间戳。

Worker容器安全风险

Worker在容器内执行任意命令，LLM可能生成恶意操作。
缓解：Scope Guard限制文件访问 + 网络隔离 + 只读挂载 + 命令白名单。

用户接受度不确定

开发者可能不信任AI产出，或觉得审查成本和自己写差不多。
缓解：初期聚焦简单重复任务(CRUD)证明价值 + 高质量Diff展示 + 逐步建立信任。

向量检索召回率不足

语义检索可能漏掉关键代码。
缓解：Layer A精确 + Layer B语义双通道 + Layer D共现分析 + reranker精排。
