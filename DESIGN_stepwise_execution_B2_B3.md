# DESIGN: 大任务分步执行 B2/B3 + 跨子任务上下文传递

> 状态：草案待拍板 | 接 v0.8.4（方案A 上下文预注入 + B1 动态 max_iter 已上线）
> 触发：task 82104bf1 真实需求虽 DONE，但 worker 仍撞 80 上限多轮迭代修复（428s）。
> 关联洞察：小模型做"手"无能力上限，瓶颈在"单次 loop 装不下多文件 + 步数预算"。

## 0. 现状澄清（走读已确认，避免重复造轮子）

- ✅ **撞上限不丢产出**：`_run_agent` 捕获 GraphRecursionError 后，已做改动仍在沙箱，后续 pull-back + L1 闸门按真实文件裁决（executor.py L1431-1437）。所以"部分产出保留"**基础已有**。
- ❌ **真瓶颈**：worker 在【单次 agent loop】里要写完所有 scope 文件，loop 内每个文件 read→think→write→自检都吃步数，多文件累加就撞上限（428s 是反复撞上限+重试的总和，不是单轮）。
- 所以 **B2 的本质不是"保留产出"（已有），而是"把一次写 N 文件拆成多阶段、每阶段聚焦、阶段间保进度"**。

## 1. 三项的依赖关系与建议顺序

```
B2（worker 内分阶段执行）        ← 独立，风险低，性价比最高 → 先做
B3（Brain 细粒度依赖序拆分）     ← 风险高（可能与垂直切片打架）→ 谨慎，需防过拆
跨子任务上下文传递              ← 依赖 B3（多子任务才有"前序"）→ 跟 B3 一起
```

**关键风险（B3）**：我们 v0.8.3 刚立"垂直切片守卫"（一个功能跨多文件归一个子任务），v0.8.4 又把 writable 上限放宽到软6硬12 来支持它。**B3 把功能拆成"接口→实现→装配"多子任务，方向与垂直切片相反**。处理不好会回到"多子任务同文件 MERGE 冲突 / 测试子任务写错 / replan 死循环"的老坑。

## 2. 渐进明细方案

### B2：worker 分阶段执行（一个子任务内）

不改 Brain 拆分，只改 worker 执行方式。当子任务 writable+create 文件数 > 阈值（如 3）时：

- **B2-a（最小，推荐先做）**：把"一次 loop 写所有文件"改成**按文件分批的多次 agent loop**。每批聚焦 1-2 个文件，写完即 pull-back commit（沙箱内 git add 该文件），下一批的 loop 从干净状态继续。每批独立步数预算，不会一次爆。
- **B2-b（增强）**：每批之间做轻量进度检查（已写文件编译过没），失败的文件下一批重试，成功的不动。

**收益**：N 文件从"1 个 100 步 loop（爆）"变成"N 个 30 步 loop（每个够用）"，且阶段间不丢进度。

### B3：Brain 细粒度依赖序拆分（防过拆）

**不是无脑拆碎**，而是：当一个垂直功能确实大（writable > 软上限 6，或 LLM 判定多组件）时，**在垂直切片内部按依赖序排子任务**：
- 接口/契约先行（如 Service 接口、DTO）→ 实现（ServiceImpl）→ 装配（Controller 调用）
- 子任务间用 `depends_on` 串成序，**串行执行**（非并行），后序能看到前序产出。
- **守卫**：只对"真的大"的功能拆（≥7 文件 或 LLM 显式标记多组件）；≤6 文件仍单子任务垂直切片（不破坏 v0.8.4 行为）。

### 跨子任务上下文传递

B3 拆出依赖序子任务后，**后序子任务的 prompt 注入前序子任务的产出 diff/新建的符号签名**：
- st-2（实现）能看到 st-1（接口）定义了什么方法签名 → 实现时对齐。
- 复用已有 `context_snippets` 机制（方案A），扩展为也注入"前序兄弟子任务的产出"。

## 4. 验证标准

- B2：导出 Excel 需求（或更大的多文件需求）worker **不再撞上限**（或撞了也因分阶段已写完大部分），总耗时显著 < 428s。
- B3+上下文：一个 ≥7 文件的真功能（如"新增一个完整 CRUD 模块"）能拆成依赖序子任务、后序对齐前序契约、端到端 DONE。

## 5. 我的核心建议
**先做 B2（独立、低风险、直接提速），用导出 Excel 重验**。B2 若已让 worker 不撞上限，B3 的必要性就降低（避免过度设计、避免破坏刚立的垂直切片）。**B3+跨子任务上下文留到 B2 验证后再评估是否需要**——用真实数据决定，不预先全做。
