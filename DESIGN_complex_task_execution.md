# DESIGN: 复杂任务执行能力 — 喂够上下文 + 分步执行 (C3/S5)

> 状态：草案待拍板 | 触发：task 34fab09e / b63792fa 真实产品经理式需求验证
> 关联洞察（用户）：**"worker 失败的真因几乎总是拆解层信息没喂到位（scope/契约/周边代码/目标）或步数耗尽"** —— 已被真实数据精确证实。

## 1. 问题（真实数据驱动，非臆测）

产品经理式需求"为系统新增操作日志导出 Excel 功能"（零文件名）连跑两次，崩溃画像：

| 观察 | 数据 | 结论 |
|---|---|---|
| 文件定位**对了** | ANALYZE 正确定位 SysOperLog/Controller/Service/Impl 4 文件 + readable 放了 ExcelUtil/Excel/BaseController | codegraph 检索能力**存在**，比预想强 |
| worker 撞迭代上限 50 **共 8 次** | LOCATING→CODING→VERIFYING 反复，202s 后空 diff | 步数预算耗尽 |
| 84 个沙箱命令**绝大多数是 cat/ls** | worker 反复读文件探索代码 | **步数全耗在"重新定位+理解"，没到"写代码"就用完** |

**根因**：worker 拿到的 scope 只有**文件路径**，没有**理解该功能所需的上下文**（每个文件的相关代码片段、参照写法、调用契约）。于是 worker 在执行阶段被迫用宝贵的 agent loop 步数去 `cat` 一堆文件重新理解——而这本该由 Brain/ELABORATE 在**拆解时一次性喂给它**。

**这不是 27B 能力上限**（用户洞察正确）——是拆解层把"定位+理解"的活甩给了执行层，执行层步数不够。

## 2. 设计目标

让一个跨多文件的完整功能，能被小模型 worker 在步数预算内**真正写完**。两条腿：
- **A（喂够上下文）**：ELABORATE 阶段把每个 scope 文件的**关键代码片段 + 参照实现 + 契约**直接注入子任务，worker 拿到即可写，不必自己 cat 探索。
- **B（分步执行）**：当单个功能确实大到一轮 loop 装不下时，支持**子任务内分阶段 checkpoint**或**按文件分步产出**，而非一次塞满 50 步就崩。

## 3. 渐进明细方案

### 方案 A：上下文预注入（ELABORATE 增强）— 优先级最高，性价比最高

ELABORATE 已会"将 Java 同 package 类纳入 readable"。增强为：
1. 对每个 writable/readable 文件，用 codegraph **抽取相关代码片段**（类签名、要调用的方法签名、参照实现的完整方法体），写进子任务的 `context_snippets` 字段。
2. worker prompt 里直接带这些片段 → worker **不需要 cat 探索**，步数全用于写代码。

**预期**：worker 的 cat/ls 命令从 ~60 个降到个位数，步数省出来写代码。

### 方案 B：分步执行（max_iterations 不够时的兜底）

- B1（最小）：complex/跨多文件任务的 `max_iterations` 从 50 提到 80~100（治标，但立竿见影）。
- B2（结构）：worker 内"按文件 checkpoint"——每写完一个文件 commit 一次中间产出，撞上限时已完成的文件不丢（部分产出 > 全空 diff）。
- B3（架构）：Brain 把"大功能"显式拆成**有依赖序的多个小子任务**（接口先行→实现→装配），每个子任务单文件级、步数充裕。这回归"垂直切片"但粒度更细。
